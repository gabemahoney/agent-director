# recov-7: delete on a live-state row bypasses guards

Per SRD §12 (`delete` is admin): the verb bypasses all state-precondition
guards. A live-state row is removed by id even though no other verb
would touch it that way.

## Setup

Per-case DB-reset. Seed a live row directly via sqlite3 (no real
spawn — we're only proving the DB cleanup contract; the orphan tmux
session that a real spawn would leave behind is out of scope here).

## Steps

1. INSERT a `waiting` row.
2. `delete --claude-instance-id <id>` → result map `{<id>: "ok"}`.
3. Row is gone.

## Pass criteria

- delete exits 0 with the result map.
- Row deleted from DB.

## Script

```bash
set -euo pipefail
sqlite3 "$HOME/.claude-director/state.db" \
  "INSERT INTO spawns (claude_instance_id, state, cwd, tmux_session_name, relay_mode)
   VALUES ('id-live-del', 'waiting', '/tmp', 'cd-live-del', 'off');"

res=$(claude-director delete --claude-instance-id id-live-del)
got=$(printf '%s' "$res" | jq -r '.results["id-live-del"]')
if [ "$got" != "ok" ]; then
    echo "FAIL: result=$got; want ok"; exit 1
fi
n=$(sqlite3 "$HOME/.claude-director/state.db" \
    "SELECT COUNT(*) FROM spawns WHERE claude_instance_id='id-live-del';")
if [ "$n" != "0" ]; then
    echo "FAIL: live-state row survived delete"; exit 1
fi
echo "recov-7 ok: delete bypasses state-precondition guard on live rows"
```
