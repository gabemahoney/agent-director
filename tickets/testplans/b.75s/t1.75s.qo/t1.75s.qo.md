# Epic 3 — `epic-03-spawn` testplan

t1 collector for Epic 3 (`t1.8op.sb` — Spawn a tracked Claude). Each t2
is a plain-English case the Docker harness executes. The Epic 3
Progression Contract requires every t2 to pass under `make test-docker
EPIC=epic-03-spawn` per the orchestrator's first-hand audit.

## Cases

| case | what it pins |
|---|---|
| `spawn-1-pending-to-waiting` | `spawn --cwd /tmp` returns claude_instance_id; immediate `status` = pending; first SessionStart hook bumps to waiting. |
| `spawn-2-prompt-to-working` | UserPromptSubmit hook synthesized via stdin drives state = working. |
| `spawn-3-ask-user-state` | PreToolUse hook with `tool_name=AskUserQuestion` drives state = ask_user. |
| `spawn-4-invalid-cwd-relative` | `spawn --cwd ./relative` → ErrCwdNotAPath on stderr; non-zero exit; no DB row. |
| `spawn-5-denied-flag-settings` | `spawn --cwd /tmp -- --settings={}` → ErrSpawnDeniedFlag. |
| `spawn-6-reserved-env-key` | `spawn --cwd /tmp --extra-env CLAUDE_DIRECTOR_FOO=bar` → ErrReservedEnvKey. |
| `spawn-7-cwd-canonicalization` | `/tmp` and `/tmp/./` both store the same canonical cwd. |
| `spawn-8-auth-env-passthrough` | `--extra-env ANTHROPIC_API_KEY=...` is accepted (not reserved) and propagates to the tmux session env. |
| `spawn-9-pretrust-cwd` | `spawn --cwd <fresh-dir>` pre-writes `projects.<cwd>.hasTrustDialogAccepted=true` into `~/.claude.json` so Claude Code skips its workspace-trust dialog (bug b.f75). |
| `spawn-10-tmux-session-name` | `--tmux-session-name <name>` end-to-end: verbatim persisted name; validation rejects reserved/control chars, > 64 bytes, explicit empty; live-collision surfaces tmux's own refusal (no `ErrTmuxSessionNameTaken`); name reuse across ended spawns works. |

## Driver mode

All cases below are written for `DRIVER_MODE=shell` — the orchestrator
runs the harness with `make test-docker EPIC=epic-03-spawn`. Each t2
body has a single fenced ` ```bash ` block the driver executes after
the per-case DB-reset fixture.

The state-transition cases (`spawn-1`, `spawn-2`, `spawn-3`) emit hook
payloads directly via `claude-director hook` rather than launching a
real Claude — this keeps the harness API-free for the gate run and
exercises exactly the SRD §5.2 mapping without any cross-contamination
from Claude Code's own internals. `DRIVER_MODE=claude` runs against a
real Claude in Epic 9's resume / Epic 10's relay testplans.

## Notes for the orchestrator

- Audit standard (SRD §17): run `make test-docker EPIC=epic-03-spawn`
  yourself, inspect the JSON stream case-by-case, confirm each `t2`
  emitted `{"status":"pass"}`.
- The state-transition cases synthesize hook payloads — no `claude`
  invocation is required. The harness should pass without any API
  credentials.
- The validation cases exercise the JSON error envelopes — pure CLI
  behavior, no DB / tmux / claude touch.

## tmux session sweep

Cases that spawn (`spawn-1`, `spawn-7`, `spawn-8`, `spawn-9`) leave
tmux sessions under the `cd-*` prefix. The DB-reset fixture kills
`cd-*` sessions between cases (see `test/driver/db-reset.sh`), so
cases never see each other's leftovers.
