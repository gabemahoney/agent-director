# Epic 11 — `epic-11-mcp` testplan

t1 collector for Epic 11 (`t1.8op.o6` — Stdio MCP server). Each t2
is a plain-English case the Docker harness executes. The Epic 11
Progression Contract requires every t2 to pass under
`make test-docker EPIC=epic-11-mcp`.

## Sourced from

- Epic ticket: `t1.8op.o6` (Acceptance Criteria #1–#3).
- SRD §3.3 (long-lived process model), §4.1 (manifest is the source
  of truth), §12.2 (JSON envelope shape), §13.1 (err_name catalog).

## Cases

| case | what it pins |
|---|---|
| `mcp-1-initialize-handshake` | initialize → response carries protocolVersion=2024-11-05, serverInfo.name=agent-director, tools capability advertised. |
| `mcp-2-tools-list-count-matches-manifest` | tools/list returns `len(manifest.Verbs) - 3` tools (filters `hook`, `serve`, `trail-emit`); drift-by-construction invariant. |
| `mcp-3-tools-list-includes-key-verbs` | tools/list includes `spawn`, `status`, `send_keys`, `read_pane`, `decide`, `find_missing`, `get_permission` (hyphens → underscores). |
| `mcp-4-tools-call-status-routes` | seed a row; tools/call name=status → JSON result has `state` field matching the seeded row. |
| `mcp-5-tools-call-error-envelope` | tools/call name=status with unknown id → MCP error response with `data.err_name = ErrSpawnNotFound`. |
| `mcp-6-100-sequential-calls` | Stream 30 sequential help calls in one server lifetime; every response is well-formed JSON-RPC; server still responsive at the end. |
| `mcp-7-unknown-method-error` | Bad JSON-RPC method → -32601 method-not-found. |
| `mcp-8-input-schema-shape` | tools/list's `spawn` entry has inputSchema.type=object and properties.cwd.type=string. |

## Driver mode

All cases are written for `DRIVER_MODE=shell` — the orchestrator runs
the harness with `make test-docker EPIC=epic-11-mcp`. Each t2 body
has a single fenced ` ```bash ` block the driver executes after the
per-case DB-reset fixture.

The cases pipe JSON-RPC requests directly into
`agent-director serve --stdio` via `printf | agent-director serve
--stdio`. Stdout (the JSON-RPC responses) is captured and parsed
with `jq`. The server exits on stdin EOF — the same mechanism Claude
Code uses when a session ends.

## Notes for the orchestrator

- Audit standard (SRD §17): run `make test-docker EPIC=epic-11-mcp`
  yourself, inspect the JSON stream case-by-case.
- The MCP server does not require a real Claude API key; it speaks
  JSON-RPC directly. The harness's pipe-based driver is the same
  shape Claude Code uses, minus the LLM client.

## tmux session sweep

These cases do NOT use `agent-director spawn` or any tmux
interaction. No `cd-*` sessions are created; the db-reset fixture's
sweep is a no-op for these cases.
