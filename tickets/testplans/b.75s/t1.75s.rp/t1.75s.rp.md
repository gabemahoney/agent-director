# Release pipeline (`release-pipeline`) testplan

t1 collector for Epic 7 of bee `b.qe2` (coordinated release pipeline:
`skills/release-agent-director/release.sh` + its supporting helpers,
`SKILL.md`, and the toolchain-pin diff script). Each t2 is a
plain-shell case the Docker harness executes. Runs under
`make test-docker EPIC=release-pipeline`; the driver
(`test/driver/run-testplan.sh`) resolves the slug by case-insensitive
substring match against this t1's frontmatter title, so the canonical
substring `release-pipeline` must remain in the title verbatim.

The cases pin the *behavior* of `release.sh` — phase ordering, the
H3 halt, the darwin/arm64 offline halt, the matrix-red halt, and
the gh-release asset list — using `--dry-run` so no irreversible
side effect is produced. Live-run behavior (`git push --tags`,
`npm publish`, `gh release create`) is operator action and is
deliberately not exercised by this testplan; the operator runs the
real release after this gate goes green.

## Driver mode

All cases use `DRIVER_MODE=shell` — the default for
`make test-docker`. Each t2 body has a single fenced ` ```bash `
block the driver executes after the per-case DB-reset fixture.

The cases need:

- `bash`, `git`, `grep`, `awk`, `sed` — present in the base harness image.
- `go` (for `make release-binaries` if a case invokes the build phase).
  Cases 1, 3, 4, 5 set `--no-build` + `AD_RELEASE_SKIP_CABI=1` so they
  exercise the script's dry-run flow without paying the build/CGO cost
  or requiring a green cabi-matrix run.
- `npm`/`bun` — case 1 invokes the script's full dry-run which runs
  `npm publish --dry-run --ignore-scripts` and `bun run version-bump`;
  these are pre-installed in the harness image per Epic 5 setup.
- A clean writable workdir per case. Each case copies the read-only
  `/work/source` bind-mount into a writable temp directory so the
  script can mutate `package.json` files (the script's EXIT trap
  rolls back the mutations in dry-run).

No case needs API auth, a real `agent-director` binary spawned via
`tmux`, or a real Claude session. The repo's read-only source
bind-mount at `/work/source` is sufficient.

## Notes for the orchestrator

- Audit standard (SRD §17): run
  `make test-docker EPIC=release-pipeline` yourself, inspect the
  JSON stream case-by-case, confirm each `t2` emitted
  `{"status":"pass"}`.
- A contrived regression demonstrates the testplan has teeth: e.g.
  removing the `phase_ok build` line from `release.sh` should fail
  `release-pipeline-1`; removing the H3 sentinel regex should fail
  `release-pipeline-2`; renaming `AD_RELEASE_SIMULATE_DARWIN_ARM64_OFFLINE`
  to a different env var should fail `release-pipeline-3`; deleting
  a CLI binary line from `release_assets()` should fail
  `release-pipeline-5`.
- The cases do not launch any tmux sessions, so they leave no `cd-*`
  sessions for the DB-reset fixture to sweep.

## tmux session sweep

No `cd-*` sessions are created. The db-reset fixture's session
sweep is a no-op for these cases.
