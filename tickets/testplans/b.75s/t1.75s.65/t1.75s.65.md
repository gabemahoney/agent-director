# Epic 10 — `epic-10-relay` testplan

t1 collector for Epic 10 (`t1.8op.g8` — Permission relay). Each t2
is a plain-English case the Docker harness executes. The Epic 10
Progression Contract requires every t2 to pass under
`make test-docker EPIC=epic-10-relay`.

## Sourced from

- Epic ticket: `t1.8op.g8` (Acceptance Criteria #1–#4).
- SRD §6.2 (relay polling), §6.3 (envelope wire shape), §6.4
  (fail-closed boundary), §6.5 (env-var read of relay mode), §12
  (decide verb), §13.1 (error catalogue).

## Cases

| case | what it pins |
|---|---|
| `relay-1-allow-roundtrip` | spawn relay=on row + hook running in background (polling) + `decide --decision=allow --reason "ok"` → hook's stdout carries the allow envelope. |
| `relay-2-deny-with-reason` | same shape, decision=deny, reason flows verbatim into the envelope. |
| `relay-3-deny-default-message` | decide deny with empty reason → envelope's message is the SRD-defined "Denied by orchestrator". |
| `relay-4-already-decided` | first decide writes; second decide on the same instance id → ErrAlreadyDecided. |
| `relay-5-relay-mode-off` | spawn relay_mode=off; `decide` → ErrRelayModeOff (no envelope, no DB write). |
| `relay-6-no-open-request` | spawn relay_mode=on but no PermissionRequest hook running → decide → ErrNoOpenPermissionRequest. |
| `relay-7-send-keys-while-relayed` | spawn relay_mode=on, state=check_permission, `send-keys` → ErrSendKeysWhileRelayed (Epic 4's stubbed guard, now active). |
| `relay-8-fail-closed-timeout` | hook with `relay.timeout_seconds = 2`; no decide arrives → fail-closed deny envelope after ~2s. |

## Driver mode

All cases are written for `DRIVER_MODE=shell` — the orchestrator runs
the harness with `make test-docker EPIC=epic-10-relay`. Each t2
body has a single fenced ` ```bash ` block the driver executes after
the per-case DB-reset fixture.

The relay cases mock the Claude Code → hook-handler boundary by
piping a synthesized PermissionRequest payload to
`agent-director hook` directly. The hook runs in the background while
the case body invokes `decide` (or doesn't, for the fail-closed
cases); the hook's stdout is captured to a tempfile and inspected
once the hook exits.

`AGENT_DIRECTOR_RELAY_MODE=on` is set in the hook's environment to
activate the relay path. The state-tracking UPSERT writes the row to
state=check_permission as a side effect; the relay-side
DELETE-INSERT writes the permission_requests row that `decide`
updates.

## Notes for the orchestrator

- Audit standard (SRD §17): run `make test-docker EPIC=epic-10-relay`
  yourself, inspect the JSON stream case-by-case.
- The polling cadence floor is 50ms (SRD §6.2). Tests that need to
  signal the hook usually sleep ≥100ms after backgrounding it before
  invoking `decide`, to ensure the row has been UPSERTed.
- `relay-8-fail-closed-timeout` uses a 2-second `relay.timeout_seconds`
  override via a per-case `~/.agent-director/config.toml` so the
  full harness runs in under a minute.

## tmux session sweep

The relay cases do NOT use `agent-director spawn` — they seed rows
directly via sqlite3 and invoke the hook handler manually. No tmux
sessions are created, so the db-reset fixture's `cd-*` sweep is a
no-op for these cases.
