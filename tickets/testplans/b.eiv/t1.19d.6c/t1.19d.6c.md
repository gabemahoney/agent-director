# subprocess-client testplan

t1 collector for the v0.5.0 subprocess-CLI Client (work bee `b.19d`,
Epic C `t1.19d.6c`). Each t2 is a plain-English case the Docker
harness executes under `make test-docker EPIC=subprocess-client`
with `DRIVER_MODE=shell` against a clean linux/x86_64 container.

The four cases mirror the in-repo `bun test` cases (SR-10.3ŌĆō10.6) but
run against an *installed-tarball* layout: the locally-packed
`agent-director-*.tgz` is `bun install`ed into a fresh `${HOME}`, the
Client is exercised through `import "agent-director"` from a tiny
driver script, and the bundled CLI binary is resolved through
`@agent-director/<platform>` exactly as a downstream consumer would.

| case | what it pins | maps to |

## Driver mode

All cases are written for `DRIVER_MODE=shell` ŌĆö the orchestrator runs
the harness with `make test-docker EPIC=subprocess-client`. Each t2
body is a single fenced ` ```bash ` block the driver executes after
the per-case `${HOME}=$(mktemp -d)` reset.

The tarball is staged into the container at a known path
(`/work/source/dist/agent-director-*.tgz`) by the harness image build;
each case `bun install`s it into a per-case `${HOME}/proj/` so no
network access to npmjs.com is required.

## Notes for the orchestrator

- Audit standard (SRD ┬¦17): run
  `make test-docker EPIC=subprocess-client` yourself; inspect the JSON

- The 4 cases use the installed-tarball context rather than the in-repo
