# Test case: DB-reset fixture isolates between cases (paired with smoke-2)

Maps to Subtask 2.3 acceptance criteria: paired case B. Smoke-2 inserted a
row; the per-case DB-reset fixture should have wiped it before this case
runs. If smoke-3 sees the row, the fixture is broken.

## Setup

This case must run *after* smoke-2 in the same `make test-docker` run. The
per-case DB-reset fixture runs between every t2 case (see
`test/driver/run-testplan.sh`); it removes `state.db` and re-runs
`agent-director help` to rebuild the current schema.

## Steps

1. Confirm `~/.agent-director/state.db` exists.
2. Confirm `PRAGMA user_version` returns 2 (fixture re-stamped the schema
   at the current version, post v1→v2 migration).
3. Confirm the `spawns` table has zero rows.
4. Confirm the `smoke-leftover` `claude_instance_id` row inserted by
   smoke-2 is **not** present.

## Pass criteria

`COUNT(*)` from `spawns` returns `0`; SELECT for `smoke-leftover` returns
no rows; exit 0.

If this case fails, the DB-reset fixture is not running between cases —
test results across the whole harness become unreliable. Stop the suite and
investigate `run-testplan.sh`'s per-case fixture invocation.

```bash
set -euo pipefail
DB="$HOME/.agent-director/state.db"
test -r "$DB"
version=$(sqlite3 "$DB" 'PRAGMA user_version')
test "$version" = "2"
count=$(sqlite3 "$DB" 'SELECT COUNT(*) FROM spawns;')
test "$count" = "0"
leftover=$(sqlite3 "$DB" "SELECT COUNT(*) FROM spawns WHERE claude_instance_id = 'smoke-leftover';")
test "$leftover" = "0"
echo "smoke-3 ok: DB clean — fixture isolated smoke-2's leftover row"
```
