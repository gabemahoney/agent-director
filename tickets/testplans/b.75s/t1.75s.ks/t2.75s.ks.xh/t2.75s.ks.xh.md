# kill-4: kill is idempotent across repeated calls

Per SRD §5 + Epic 5 t2.31 ticket: kill is idempotent. The verb does
NOT flip the row's state (find-missing reconciles that out-of-band in
Epic 8), so a second call on the same id must still return 0 — the
inner tmux call is now a swallowed no-op against an already-gone
session.

## Setup

Per-case DB-reset. A `cat` pane under `cd-k4`.

## Steps

1. Start `tmux new-session -d -s cd-k4 -- cat`.
2. Seed `spawns` row at state `waiting`, session_name `cd-k4`.
3. First kill → session gone.
4. Second kill against the same id → still exits 0.

## Pass criteria

- Both kills exit 0.
- `tmux has-session -t cd-k4` returns non-zero after the first kill.

## Script

```bash
set -euo pipefail
tmux new-session -d -s cd-k4 -- cat
sqlite3 "$HOME/.agent-director/state.db" \
  "INSERT INTO spawns (claude_instance_id, state, cwd, tmux_session_name, relay_mode)
   VALUES ('id-k4', 'waiting', '/tmp', 'cd-k4', 'off');"
agent-director kill --claude-instance-id id-k4 >/dev/null
if tmux has-session -t cd-k4 2>/dev/null; then
    echo "FAIL: session not gone after first kill"; exit 1
fi
agent-director kill --claude-instance-id id-k4 >/dev/null
echo "kill-4 ok: idempotent across repeated calls"
```
