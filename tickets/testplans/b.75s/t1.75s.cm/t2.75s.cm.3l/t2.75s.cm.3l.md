# ci-matrix-2-two-legs-with-expected-runs-on

Pins Epic 6 AC: the workflow declares exactly two matrix legs
`linux-amd64` and `darwin-arm64`, whose `runs-on` values are
respectively the full self-hosted label tuples
`[self-hosted, Linux, X64, linux-amd64]` and
`[self-hosted, macOS, ARM64, darwin-arm64]`. Bare `self-hosted`
is intentionally forbidden — that would let the workflow route to
a different self-hosted runner the operator may add later.

`darwin-amd64` was dropped from the v1 platform set on 2026-05-24
(no Intel Mac users to serve; GH-hosted macos-13 carries a 10x
billing multiplier). This case actively asserts that
`name: darwin-amd64` is **absent** from the workflow so the drop
cannot regress.

## Setup

No DB state. The check is a pure inspection of the committed file at
`/work/source/.github/workflows/cabi-matrix.yml`.

## Steps

1. Assert the workflow file exists.
2. Assert each of the two `name:` declarations is present.
3. Assert `name: darwin-amd64` is **absent**.
4. Pair each `name:` with its immediately-following `runs-on:` (within
   the next 5 lines, to absorb intervening comments) and verify the
   declared label tuple.
5. Spot-check that both legs' `runs-on:` declares all four labels in
   their respective conjunctive tuples.
6. Emit pass marker.

## Pass criteria

- `name: linux-amd64` is paired with a `runs-on:` line referencing all
  four of `self-hosted`, `Linux`, `X64`, and `linux-amd64`.
- `name: darwin-arm64` is paired with a `runs-on:` line referencing
  all four of `self-hosted`, `macOS`, `ARM64`, and `darwin-arm64`.
- `name: darwin-amd64` does NOT appear anywhere in the workflow.
- Final line matches `ci-matrix-2 ok: ...`.

## Script

```bash
set -euo pipefail

YML="/work/source/.github/workflows/cabi-matrix.yml"
if [[ ! -f "$YML" ]]; then
    echo "FAIL: $YML not found"
    exit 1
fi

# darwin-amd64 must be gone (v1 scope shrink, 2026-05-24).
if grep -qE '^[[:space:]]*-?[[:space:]]*name:[[:space:]]*darwin-amd64' "$YML"; then
    echo "FAIL: darwin-amd64 leg should not be present in the v1 workflow"
    exit 1
fi

# Each leg has a `- name: <leg>` line followed by a `runs-on:` line
# within 5 lines. We capture the runs-on line that pairs with each
# name and assert its contents.

pair_runs_on() {
    local name="$1"
    grep -A 5 -F "name: ${name}" "$YML" | grep -E '^[[:space:]]*runs-on:' | head -n 1
}

linux_runs=$(pair_runs_on linux-amd64)
for label in "self-hosted" "Linux" "X64" "linux-amd64"; do
    if [[ "$linux_runs" != *"$label"* ]]; then
        echo "FAIL: linux-amd64 runs-on missing required label '$label': ${linux_runs:-<empty>}"
        exit 1
    fi
done

darm_runs=$(pair_runs_on darwin-arm64)
for label in "self-hosted" "macOS" "ARM64" "darwin-arm64"; do
    if [[ "$darm_runs" != *"$label"* ]]; then
        echo "FAIL: darwin-arm64 runs-on missing required label '$label': ${darm_runs:-<empty>}"
        exit 1
    fi
done

# Belt-and-suspenders: a bare `runs-on: self-hosted` on either leg
# would slip past the per-label loop above (every substring matches
# the unquoted form). Verify the runs-on is a bracketed list, not a
# bare scalar.
for tuple_label in "linux-amd64:$linux_runs" "darwin-arm64:$darm_runs"; do
    leg="${tuple_label%%:*}"
    line="${tuple_label#*:}"
    if [[ "$line" != *"["* || "$line" != *"]"* ]]; then
        echo "FAIL: $leg runs-on not declared as a label-tuple list: ${line:-<empty>}"
        exit 1
    fi
done

echo "ci-matrix-2 ok: two legs with expected runs-on values"
```
