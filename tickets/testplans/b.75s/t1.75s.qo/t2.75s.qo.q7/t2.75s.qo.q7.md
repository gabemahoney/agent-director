# spawn-8: auth env var passes validation and reaches the row

Maps to SRD §14.4: `ANTHROPIC_API_KEY` and `CLAUDE_CODE_OAUTH_TOKEN`
are explicitly allowed in `extra_env` — they do NOT carry the reserved
`CLAUDE_DIRECTOR_*` prefix. The harness can verify validation accepts
the value without actually authenticating against Anthropic.

This is the empirical-research-driven case (see
`reference/anthropic-api-key-auth-research.md` and
`reference/max-account-auth-research.md`). The Docker harness in
`DRIVER_MODE=claude` exercises the real round-trip with a test
secondary account; the `DRIVER_MODE=shell` case here pins the
validation surface only.

## Setup

Per-case DB-reset has wiped the DB.

## Steps

1. `claude-director spawn --cwd /tmp --extra-env ANTHROPIC_API_KEY=sk-ant-test`
   → exit 0; capture `claude_instance_id`.
2. `claude-director get --claude-instance-id <id>` → row exists with
   state = `pending`.

The pin is "validation didn't reject the auth env var" (a value
matching `CLAUDE_DIRECTOR_*` would have hit `ErrReservedEnvKey`; the
fact that this spawn succeeds proves the carve-out works).

## Pass criteria

- Exit 0 from the spawn call.
- Returned `claude_instance_id` is non-empty.
- `get` returns the row in `pending` state.

## Script

```bash
set -euo pipefail
id="$(claude-director spawn --cwd /tmp --extra-env ANTHROPIC_API_KEY=sk-ant-test \
        | jq -r '.claude_instance_id')"
test -n "$id"
state="$(claude-director get --claude-instance-id "$id" | jq -r '.state')"
test "$state" = "pending"
echo "spawn-8 ok: auth env var accepted; id=$id state=pending"
```
