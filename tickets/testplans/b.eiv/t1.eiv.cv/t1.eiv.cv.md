# subprocess-coverage testplan

t1 collector for the subprocess-specific surface that the existing
`subprocess-client` slug (`t1.19d.6c`) does not cover. Each t2 below
runs against the *installed-tarball* layout — `bun install`ed from a
locally-packed `agent-director-*.tgz`, then exercised via
`import { Client } from "agent-director"` — exactly the boundary
v0.5.0 ships to consumers.

Invocation: `make test-docker EPIC=subprocess-coverage`. Driver mode
is `shell` (each t2 body is a single fenced `bash` block).

## Sourced from

- Work bee: `b.19d` / SRD `t1.eiv.ag`.
- Specifically: SR-1.4 (env-var inheritance), SR-4 (typed-error
  catalog parity), SR-5 (consumer-signal propagation), SR-6 (per-call
  timeout), SR-7 (large stdout/stderr drain).

## Cases

| case | what it pins | maps to |
|---|---|---|
| `subprocess-coverage-1-signal` | consumer-delivered SIGTERM on the in-flight subprocess surfaces as `ErrConsumerSignal`; the Client queue is not wedged (a follow-up call succeeds). | SR-5.2 / SR-5.4 / SR-3.3 |
| `subprocess-coverage-2-timeout` | `callTimeoutMs=500` against a CLI that won't exit fires SIGTERM → 2 s grace → SIGKILL; the call rejects with `ErrCallTimeout`. | SR-6.1 / SR-6.2 / SR-6.5 |
| `subprocess-coverage-3-large-stdout` | a CLI binary that prints ~200 KB of JSON on stdout (well past pipe-buffer size) does not deadlock; the Client parses it and returns the envelope. | SR-7.1 |
| `subprocess-coverage-4-env-inherit` | env vars set on the consumer process are visible inside the spawned subprocess (proven by echoing them back through the result envelope). | SR-1.4 |
| `subprocess-coverage-5-error-catalog` | every catalog entry in `pkg/api/errnames/catalog.json` round-trips through the Client as `instanceof Err<Name>`. | SR-4.1 / SR-4.2 / SR-4.4 |

## Driver mode

All cases are `DRIVER_MODE=shell` and self-contained inside a single
fenced ` ```bash ` block. The shared tarball helper from `t1.19d.6c`
(`/work/source/tickets/testplans/b.eiv/t1.19d.6c/build-tarball.sh`)
is sourced so the tarball cache + `stage_platform_pkg` helper are
reused without duplication. Each case then replaces the staged
platform binary at
`${PROJ}/node_modules/@agent-director/linux-x64/bin/agent-director`
with a tiny shell stub that simulates the behavior under test —
slow, signal-killed, huge-stdout, env-echoing, or error-envelope
emitting — and runs its `driver.mjs` against that stub.

The stub-binary pattern is deliberate: the surfaces covered here are
properties of the Client / subprocess plumbing (env snapshot,
timeout, signal handling, drain, error-map routing), not properties
of the CLI binary's verb implementations. Decoupling the Client
contract from the CLI's internal verb behavior is exactly what the
subprocess boundary lets us do; these cases lean on that.

## Notes for the orchestrator

- Audit standard: run `make test-docker EPIC=subprocess-coverage`
  yourself; inspect the JSON stream case-by-case.
- These cases must NOT touch `t1.19d.6c` (parallel worker owns it).
- The container is always linux/x86_64 so the platform sub-package
  is always `@agent-director/linux-x64`; no platform gating needed.
