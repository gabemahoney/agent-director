# Epic 2 — harness-smoke testplan

Self-referential testplan: proves the gate mechanism itself functions. Each
t2 case is a plain-English check the driver executes inside the harness
container. If every case passes, the harness is operational and every
subsequent Epic's progression contract can rely on it.

## Sourced from

- Epic ticket: `t1.8op.v9` (Docker test harness)
- SRD §15.2 (Docker integration tests, DB-reset fixture)
- SRD §17 (orchestrator-in-the-loop gate)

## Cases

| case | what it proves |
|---|---|
| smoke-1 | Driver can enumerate t2 case files for this testplan from inside the container; pinned `claude` version is on PATH; `claude-director help` exits 0 against a clean DB. No driver-Claude spawn required for the gate. |
| smoke-2 | DB-reset fixture left a clean schema-v1 DB; a row is inserted into `spawns` via `sqlite3`. Used as the precondition the next case checks. |
| smoke-3 | After the per-case DB-reset, the `spawns` table is empty — proving the fixture isolates cases. If the fixture is removed, this case fails. |
| smoke-4 | A deliberately-broken `db-reset.sh` (in a sandbox harness) causes the driver to report the case as `fail` with a `db-reset` reason — regression net for `b.emq` so the swallowed-failure path cannot return. |

## Driver mode

Default `DRIVER_MODE=shell` — Epic 2's gate runs without an API key so CI
can run on every PR. The same testplan also runs in `DRIVER_MODE=claude`
for credentialed runs (orchestrator opts in by exporting
`ANTHROPIC_API_KEY` or `CLAUDE_CODE_OAUTH_TOKEN` before `make test-docker`),
which exercises the full driver-Claude path. The cases are written in
plain-English with a single ` ```bash ` block per case so both modes work.

## Notes for the orchestrator

- The audit standard (SRD §17) is first-hand: run `make test-docker EPIC=harness-smoke`
  yourself, read the JSONL stream on stdout, confirm each case emitted a
  `{"status":"pass"}` line. The summary line tallies pass/fail/total.
- Smoke-2 and smoke-3 are a paired pair — if the fixture were removed,
  smoke-3 would fail. Verify this manually by setting `DB_RESET_DISABLE=1`
  or by removing the `bash "$DB_RESET"` call from `run-testplan.sh` and
  re-running.
